    <article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-using-unix-command-line-utilities" class="anchor" aria-hidden="true" href="#using-unix-command-line-utilities"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Using UNIX command line utilities</h1>

<h2><a id="user-content-section-1-file-system-analytics" class="anchor" aria-hidden="true" href="#section-1-file-system-analytics"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Section 1: File System Analytics</h2>

<h3><a id="user-content-navigating-the-file-system" class="anchor" aria-hidden="true" href="#navigating-the-file-system"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Navigating the file system</h3>
<p>If you've used the terminal on a Linux or UNIX system before, you're probably
very familiar with <code>ls</code> and <code>cd</code>. <code>ls</code> has some great extended options, but
accessing the results of <code>ls</code> programmatically can be difficult, and tasks like
getting access to filenames recursively can be tricky.</p>
<h4><a id="user-content-find" class="anchor" aria-hidden="true" href="#find"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>find</h4>
<p><code>find</code> provides a much more flexible interface to listing files that match a
particular criteria than ls. In particular, you can easily restrict yourself to
a particular type of file (e.g. directories or plain files), recurse
indefinitely or to some maximum depth, dispatch an action on file names that
match your search, or pair the results of find with a command like <code>xargs</code> to do
something further with files that match your filters.</p>
<p>For example, the command <code>find /usr/bin -type f -name 'py*'</code> will tell find to
list the names of all ordinary files (not directories) in <code>/usr/bin</code> whose names
start with "py".</p>
<h4><a id="user-content-du" class="anchor" aria-hidden="true" href="#du"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>du</h4>
<p>Understanding what files/directories are using space on the file system is a
problem that people have been dealing with since UNIX was invented. According to
the man page for <code>du</code>, "A <code>du</code> command appeared in Version 1 AT&amp;T UNIX."</p>
<p><code>du</code> allows you to report usage statistics of files and directories in your
system. By default, it reports its usage statistics at the block level, but the
<code>-h</code> option provides this information in a human-readable format. It's worth
noting that the results of <code>du</code> are not exact - instead sizes are reported as
multiples of block size.</p>
<p>Try running <code>du -hs *</code> in your home directory to see where the space is going.</p>
<p>Now, think about how you could output the results of <code>find</code> into <code>du</code> to do reporting on a subset of files.</p>
<h3><a id="user-content-sorting" class="anchor" aria-hidden="true" href="#sorting"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Sorting</h3>
<p>The <code>sort</code> command is pretty powerful. In most systems, it is implemented via an
external merge sort - this makes it possible to sort really huge files, even
ones that don't fit into memory, in a reasonable amount of time.</p>
<p>By default, <code>sort</code> will sort items lexicographically, but with the <code>-n</code> flag it
will sort in numeric order. This means that 10100 will come after 2 in numeric
sort order, even though it precedes 2 lexicographically. In recent versions of
GNU <code>sort</code>, the command also takes a <code>-h</code> parameter, which allows you to sort
human-readable numbers (like file sizes produced by du -h).</p>
<p>Additionally, <code>sort</code> will let you select exactly which fields of a file you want
to sort on (by default, it's the whole line), and in which order these fields
should be sorted.</p>

<h3><a id="user-content-downloading-data-files" class="anchor" aria-hidden="true" href="#downloading-data-files"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Downloading data files</h3>
<p>The first step in data analyis is typically downloading data files that you need
to process. While typically you might be used to downloading files using your web
browser, this gets tedious when you want to download many files or if you want to
automate the process. There are UNIX command line tools which you can use in
such cases to download files</p>
<h4><a id="user-content-curl" class="anchor" aria-hidden="true" href="#curl"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>curl</h4>
<p><code>curl</code> is a commonly used tool for downloading files. It is typically available
on most UNIX machines and should be installed in your VM. To download a file
using curl you need to run <code>curl &lt;url&gt; -o &lt;filename&gt;</code>. For example to download
the course webpage in html you can run something like</p>
<pre><code>curl http://biddata.github.io/datascience-fa14/index.html -o index.html
</code></pre>
<p>This will download the course webpage and save it to a file named <code>index.html</code>
in your current directory. This is of course a simple example of how you can use
<code>curl</code>. You can find other options using and features in the curl manpage using
<code>man curl</code>.</p>
<p>By default <code>curl</code> does not follow redirects when downloading a file. To make it follow
redirects, add <code>-L</code> to the curl command-line.</p>
<h4><a id="user-content-wget" class="anchor" aria-hidden="true" href="#wget"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>wget</h4>
<p>Another popular command line tool used for downloading data is <code>wget</code>. You might
have seen this used in other examples or used it before. For simple use cases
you can use either tool and we will use <code>curl</code> for this exercise. A more detailed
comparison of the two tools can be seen at
<a href="http://daniel.haxx.se/docs/curl-vs-wget.html" rel="nofollow">curl vs wget</a>.</p>
<h3><a id="user-content-http-logs-dataset" class="anchor" aria-hidden="true" href="#http-logs-dataset"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>HTTP Logs Dataset</h3>
<p>For this exercise we will be using HTTP access logs from the 1998 Soccer
WorldCup website. The <a href="http://ita.ee.lbl.gov/html/contrib/WorldCup.html" rel="nofollow">complete
dataset</a> contains
around 1.3 billion requests, and we will use a subset of it for this exercise.
As a first step download the sample dataset using <code>curl</code> from
<code>https://raw.github.com/biddata/datascience-fa14/master/lab1/data/wc_day6_1_log.tar.bz2</code> .
(You will need <code>-L</code> since this redirects to a non-github.com domain for
<a href="https://developer.github.com/changes/2014-04-25-user-content-security/">security reasons</a>.)</p>
<p>The dataset has been compressed to make the download finish faster. To get the
raw data unzip the downloaded file by running <code>tar -xf &lt;filename&gt;</code>. (Note: <code>tar</code>
is also a very frequently used command line tool and you can learn more about it
with <code>man tar</code>).</p>
<p>Having extracted the file, take a look at how the file looks by running <code>less wc_day6_1.log</code>. This will show you the first few lines of the file and you can
page through the file using the arrow keys. You will notice that each hit or
access to the website is logged as in a new line in the log file. The format of
each line is in the <a href="https://en.wikipedia.org/wiki/Common_Log_Format" rel="nofollow">Common Log File
Format</a> and this format is supported
by most HTTP servers. In this case the data has been annonymized and lets take a
look at one line from the file to explain each field</p>
<pre><code>57 - - [30/Apr/1998:22:00:48 +0000] "GET /english/images/lateb_new.gif HTTP/1.0" 200 1431
</code></pre>
<p>In the above line <code>57</code> refers to a <code>clientID</code>, a unique integer identifier for the
client that issued the request. While this field typically contains the IP
address, for privacy reasons it has been replaced with an integer. The next two
fields are <code>-</code> and say that the fields are missing from the log entry. Again
these correspond to <code>user-identifier</code>, <code>userid</code> and have been removed for
privacy reasons.</p>
<p>The next field is the time at which the request was made and this is followed by
the HTTP request that was made. In this example a GET request was made for
<code>lateb_new.gif</code>. The next field is the HTTP return code and in this example it
is 200. You can find a list of codes and their meanings from
<a href="http://www.w3.org/Protocols/rfc2616/rfc2616-sec10.html" rel="nofollow">w3</a>. The last field is
the size of the object returned to the client, measured in bytes.</p>

<p>Before we get to the exercises, lets explore the dataset and try out some basic
commands.</p>
<p>First up lets count how many visits the website got. To do this we just count
the number of lines in the file by running <code>wc -l wc_day6_1.log</code>. Your output
should look like</p>
<pre><code>1193353 wc_day6_1.log
</code></pre>
<p>We can do something more interesting by finding out how many times the ticketing
webpage was visited. To do this you could run</p>
<pre><code>grep tickets wc_day6_1.log | wc -l
29818
</code></pre>
<p>However the above line counts images and other elements which have the word
<code>tickets</code> in their path. (Note: You can verify this using <code>less</code>). To restrict
it to just <code>html</code> pages, you can use a regular expression</p>
<pre><code>grep "tickets.*html" wc_day6_1.log | wc -l
2776
</code></pre>
<p>We can also prune the dataset to only look at interesting parts of it. For
example we can just look at the first 50 URIs and their sizes using the <code>head</code>
and <code>cut</code> command.</p>
<pre><code>head -50 wc_day6_1.log | cut -d ' ' -f 7,10
</code></pre>
<p>In the above command the <code>-d</code> flag denotes what delimiter to use and <code>-f</code> stats
what fields should be selected from the line. Try out different delimiter and
field values to see how <code>cut</code> works.</p>
<p>Finally we can see how many unique URIs are there in the first 50 visits. To do
this we could run something like</p>
<pre><code>head -50 wc_day6_1.log | cut -d ' ' -f 7 | sort | uniq | wc -l
</code></pre>
<p>Here we use the tool <code>uniq</code> to only count unique URIs. Note that the input to
<code>uniq</code> should be sorted, so we use <code>sort</code> before calling <code>uniq</code>. The <code>uniq</code> tool
can also be used to count how many times an item occurs by passing it the <code>-c</code> flag.
For example if we run the same command as above but with <code>uniq -c</code> we'll get</p>
<pre><code>head -50 wc_day6_1.log  | cut -d ' ' -f 7 | sort | uniq -c | tail -10

<h3><a id="user-content-regular-expressions" class="anchor" aria-hidden="true" href="#regular-expressions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Regular expressions</h3>
<p>To do this, we will take advantage of <em>regular expressions</em>.
We have prepared a short introduction to
<a href="https://bcourses.berkeley.edu/courses/1267848/pages/regex" rel="nofollow">regular expressions</a> that
might be useful if you want a quick overview. More detail is easy to find online, for
example, in the <a href="https://www.gnu.org/software/sed/manual/sed.html#Regular-Expressions" rel="nofollow">GNU sed Manual</a>.</p>
<h3><a id="user-content-regular-expression-substituion-in-esd" class="anchor" aria-hidden="true" href="#regular-expression-substituion-in-esd"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Regular expression substituion in esd</h3>
<p><code>sed</code> (short for "Stream EDitor") is a tool for modifying and extracting data from
text files. Many sed commands were discussed in this week's reading, but, by far,
the most common sed command is regular expression substitution. You can run a substitute
command as follows:</p>
<pre><code>cat in.txt | sed 's/regexPattern/replacementString/flags' &gt; out.txt'
</code></pre>
<p>The flags are described in <code>man sed</code>, but the most notable flag is <code>g</code>, which causes sed to
find and replace all instances of the pattern rather than the first one.</p>
<p>For example,</p>
<pre><code>echo "The quick brown fox jumps over the lazy dog." | sed 's/[tT]he [a-z]*/The yellow/'
</code></pre>
<p>matches only <em>The quick</em>, resulting in
<code><em>The yellow</em> brown fox jumps over the lazy dog.</code>, while</p>
<pre><code>echo "The quick brown fox jumps over the lazy dog." | sed 's/[tT]he [a-z]*/The yellow/g'
</code></pre>
<p>matches both <em>The quick</em> and <em>the lazy</em> and results in
<code><em>The yellow</em> brown fox jumps over <em>The yellow</em> dog.</code></p>
<h3><a id="user-content-removing-cruft-in-sed" class="anchor" aria-hidden="true" href="#removing-cruft-in-sed"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Removing cruft in sed</h3>
<p>As a first step toward converting it to CSV format, let's start by removing cruft from dates in
the log file. Each date is surrounded by a <code>[</code>
and <code> +0000]</code>. We can remove the <code> +0000]</code>s using <code>sed 's/ +0000]//'</code>. To try this
on the first ten lines of the log file, use:</p>
<pre><code>head wc_day6_1.log | sed 's/ +0000]//'
</code></pre>
<p>Similarly, we can remove <code>[</code> using <code>s/\[//</code>. (We need the backslash before the <code>[</code> to prevent
sed from treating it as a special character in its regular expression syntax.) We can combine this
our previous cleaning using:</p>
<pre><code>head wc_day6_1.log | sed 's/ +0000]//' | sed 's/\[//'
</code></pre>
<p>or, only invoking <code>sed</code> once:</p>
<pre><code>head wc_day6_1.log | sed 's/ +0000]//; s/\[//'
</code></pre>
<h3><a id="user-content-backreferences-in-sed" class="anchor" aria-hidden="true" href="#backreferences-in-sed"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Backreferences in sed</h3>
<p><code>sed</code> supports <em>backreferences</em> in substitutions, allowing you to include part of the matched text
in the replacement string. To use backreferences, make a <em>capturing group</em> by surrounding part of
the matched text with <code>\(</code> and <code>\)</code>; then in the replacement text, use <code>\1</code> to place the text of
the first group, <code>\2</code> for the second and so on. For example:</p>
<pre><code>echo "The quick brown fox jumps over the lazy dog." | sed 's/\([Tt]he\) \([a-z]*\)/\1 "\2"/g'
</code></pre>
<p>results in <code>The "quick" brown fox jumps over the "lazy" dog.</code></p>
